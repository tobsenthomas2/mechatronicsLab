Lab #1: Gray Area

In this two-week exercise, you will write code which runs a motor and measures where it is, and then you’ll encapsulate that code in Python classes. You will get practice documenting code with Doxygen, testing your code thoroughly to ensure that it is reliable, and managing source code files using Git.

In the previous exercise, you used PWM to control the brightness of a light. In this lab, you’ll use the same technique to control a motor, and you’ll also measure its motion.

Encoders

A common way to measure the movement of a motor is with an optical encoder. Because most encoders can send out signals at high frequency (sometimes up to 1MHz or so), we can’t read one with regular code; we need something faster, such as interrupts or dedicated hardware. Our STM32 processors have timer/counters which can read pulses from encoders and count distance and direction of motion. Encoders send digital signals over two wires in Gray Code, or quadrature, as shown below; the two signals are about 90° out of phase:

Each time the state (A, B) of the encoder output changes, it is an indication that the encoder has moved one “tick.” The particular state change tells the direction in which the encoder has moved: for example, (0, 0) to (0, 1) indicates movement to the right on the graph above, while (0, 1) to (0, 0) indicates movement to the left. The motors we use in the lab have built-in encoders and are wired as shown below:

Color Function
Blue Encoder channel B
Yellow Encoder channel A
Red Encoder 5V supply
Black Encoder ground
Orange Motor power
Green Motor power
(None) No connection
Warning: Never allow encoder wires to contact motor power voltages.

Encoder Reading Procedure

In this part of the exercise, you’ll use the encoder on the back of a motor to measure the motor’s rotation in units of encoder ticks.

  • Disconnect the USB cable from your MicroPython board to turn off its power.

  • Connect the encoder power and ground leads to the GND and 3V3 leads on the Shoe Board’s screw terminals, then connect USB to the Shoe. (Note: Although the encoder is specified for 5 volts, it has been tested and found to work acceptably with 3.3V.) Use an oscilloscope or logic analyzer to verify that when you rotate the motor by hand, the encoder produces quadrature signals on Channels A and B.

  • Disconnect the USB cable, then connect encoder channels A and B to pins B6 and B7 of the Shoe Board’s screw terminal block. Reconnect USB and write code that sets up a timer and its CH1 and CH2 channels in encoder counting mode. Only one timer will work with those pins; check the STM32L476 datasheet. Test your code by rotating the motor by hand and verifying that the numbers from timer_N.counter() make sense.

  • Unplug USB and move the encoder’s channels A and B to pins C6 and C7, then write code which sets up another timer to read an encoder using those pins. Note that you shouldn’t use Timer 3 or Timer 5 because those are needed for the motor PWM.

Motor Control

Your ME405 kit includes an L6206 motor driver “shield”, a circuit board which can be attached atop the Nucleo and used to control two DC motors. It has a dual H-Bridge chip which can control the magnitude and direction of the torque produced by each of two motors.

An H-bridge motor driver has logic inputs to control which transistors in the bridge are turned on at any given time and to ensure that the two transistors in each half-bridge are never turned on at the same time, as this would short power to ground and let lots of smoke out. An excerpt from the L6206 motor driver IC’s datasheet is shown below:

One of the chip’s two H-bridges consists of the four N-channel FET’s in the right half of the diagram, with the motor connected to pins OUT1A and OUT2A. The microcontroller controls pins ENA, IN1A, and IN2A. A common way of using these pins is to set ENA high to enable the motor, set IN1A low, and send a PWM signal to IN2A to power the motor in one direction; reverse the signals to IN1A and IN2A to power the motor in the other direction. A truth table in the L6206 datasheet specifies the logic between these pins and what the chip does to the motor.

L6206 Pin CPU Pin Timer Ch. L6206 Pin CPU Pin Timer Ch.
ENA/OCDA PA10 - - ENB/OCDB PC1 - -
IN1A PB4 TIM3 1 IN1B PA0 TIM5 1
IN2A PB5 TIM3 2 IN2B PA1 TIM5 2

The table above shows the connections between the logic input pins on the L6206 and the CPU pins on the Nucleo motor control expansion board. The ENx and OCDx pins are connected together on the motor driver board, so it is best to set up the microcontroller pins connected to ENx/OCDx as open-drain outputs with pullup resistors enabled so that the microcontroller can read the pins to detect motor driver problems. The INnx pins can be set up as regular push-pull outputs.

Motor Control Procedure

  • Disconnect the USB cable from your MicroPython board to turn off its power. Attach a blue IHM04A1 motor driver board atop the Nucleo. Connect a DC motor to the Motor A screw terminals on the driver board with the orange and green wires which are twisted together (if they have become un-twisted, twist them loosely back together). Use the long red and black twisted pair wire from your kit to connect power from a benchtop supply to the motor driver board’s GND and Vin screw terminals.

    Double-check that only the orange and green motor leads are connected from your motor to the motor driver board. Re-attach the USB cable to the Nucleo and power the motor driver with 12 volts and a 0.5 amp current limit.

  • Now use what you have learned so far to write code that makes the motor run in each directions and stop. It is recommended to use two PWM channels, one for each INnA pin on the motor driver chip: to push the motor in one direction, set IN1A to 0% duty cycle and use IN2A other to control torque; to push the other way, set IN2A to 0% and use IN1A for control.

    Note: Unlike LEDs, the motor drivers use regular non-inverted PWM logic, so set up your timer channels in mode pyb.Timer.PWM mode, not PWM_INVERTED mode.

A Touch of Class

You’ll be using motor and encoder drivers for projects throughout this course, so it will save time to write them as classes. You should write a motor driver class which drives one motor; to use two motors, you can create two objects from that class. Similarly, one encoder class can be used to make two objects that read two encoders.

Motor Driver Class

  • Write a motor driver class called MotorDriver in a file called motor_driver.py. The constructor __init__() should set up pins to control the motor, and a method called set_duty_cycle() should set the duty cycle and direction of the motor power voltage. The class and each method must have a docstring that describes what it does and how to use it, and each parameter (except self) must have a @param descriptor. A template for the class is below. 

    class MotorDriver:
    """! 
    This class implements a motor driver for an ME405 kit. 
    """

    def __init__ (self, en_pin, in1pin, in2pin, timer):
        """! 
        Creates a motor driver by initializing GPIO
        pins and turning off the motor for safety. 
        @param en_pin (There will be several pin parameters)
        """
        print ("Creating a motor driver")

    def set_duty_cycle (self, level):
        """!
        This method sets the duty cycle to be sent
        to the motor to the given level. Positive values
        cause torque in one direction, negative values
        in the opposite direction.
        @param level A signed integer holding the duty
               cycle of the voltage sent to the motor 
        """
        print (f"Setting duty cycle to {level}")

  • Test your MotorDriver with code such as the following:

    moe = MotorDriver (a_pin, another_pin, a_timer)
moe.set_duty_cycle (-42)

  • Write test code in motor_driver.py which exercises your motor driver and exorcises any bugs. It should be in an if __name__ == '__main__' block to allow you to do the next step easily.

  • Verify that you can import motor_driver.py and operate the motor from the REPL by using your motor controller class.

Encoder Reader Class

  • Write a class which encapsulates the operation of the timer to read encoders. Your class should have the following methods:

    • A constructor __init__() which does the setup, given appropriate parameters (probably some pins and a timer)

    • A method read() which returns the current position

    • A method zero() which sets the count to zero at the current position

    • Other methods if needed

    A user should be able to create at least two objects of this class, one for each of the two encoder readers you set up in steps 2 and 3. These should be able to work at the same time.

  • After how many counts will the timer overflow? Might such overflow be a problem if your encoder reader is used to control a motor’s position? A way to solve the problem is to read the timer at least several times per second, compute the distance the encoder has moved since the previous reading, and add the distance to a Python int variable which holds the position. This int should count total movement and not reset for each revolution.

    Your code must work if the timer overflows (counts above 216 − 1) or underflows (counts below zero). Modify your code so the total position moved is correct even in case of over- and underflows.

  • Put the encoder class in a file encoder_reader.py.

Testing

  • Test your motor driver class by sending it a range of duty cycles, both positive and negative, and checking that the motor moves both ways. Make sure to check “edge cases” such as the maximum and minimum possible duty cycles for proper operation.

  • Test your encoder class, both turning the motor by hand and running the motor under power. Does your code return reasonable results when you move the motor a revolution or two one way, then back? Does it work when the motor is moving quickly?

  • Write test code in blocks with if __name__ == "__main__" in each of your module files.
